# 白泽项目介绍

## 项目定位

白泽是一个面向个人和团队工作流的本地 AI 工作中枢。它的目标不是只提供一个聊天窗口，而是把本地 Claude Code、项目记忆、逻辑规则、插件调用、权限确认、客户端更新和桌面交互整合起来，让 AI 能围绕真实工程和业务流程持续工作。

项目采用“本地客户端执行 + 服务端同步控制”的思路：

- 客户端负责用户交互、本地 Claude Code 调用、本机文件与插件桥操作。
- 服务端负责配置下发、版本更新、插件/记忆/逻辑数据同步和统一接口。
- 重要写操作通过客户端确认卡进行授权，避免 AI 静默修改外部系统。

## 为什么做这个项目

普通 AI 聊天工具通常只解决“回答问题”，但实际工作中还需要：

- 读取和理解本地工程文件。
- 维护长期记忆和项目规则。
- 调用 Jira、企业微信等外部系统。
- 对写操作做权限确认和审计。
- 让桌面客户端开箱可用。
- 支持团队内配置、插件和记忆同步。

白泽把这些能力放进同一个可本地部署的体系中，适合用来探索 AI 工程助手、需求处理助手、Jira 工作流助手、个人知识中枢和团队内部智能工具。

## 核心模块

### 1. 桌面客户端

桌面客户端位于：

```text
client/desktop
```

主要职责：

- 提供聊天界面。
- 管理会话列表。
- 支持拖拽文件和粘贴图片。
- 显示 Claude Code 执行状态。
- 显示 Jira 创建确认卡、恢复卡和结果卡。
- 调用本机 Claude Code。
- 执行本地插件桥操作。
- 接收服务端下发的运行配置。
- 支持 Windows 自动更新。

### 2. 服务端

服务端位于：

```text
src
```

主要职责：

- 提供 `/health` 健康检查。
- 提供聊天、配置、插件、附件、客户端更新等接口。
- 管理服务端配置。
- 保存和同步记忆、逻辑、审计和运行状态。
- 给客户端下发 Claude Code、Jira、插件权限等运行配置。

### 3. Claude Code 集成

Claude Code 在白泽中承担本地智能执行角色。典型能力包括：

- 读取本机工程文件。
- 分析上传文件。
- 根据用户意图决定是否写入记忆或调用插件。
- 在用户确认后执行 Jira 创建流程。
- 根据插件返回结果继续修正字段、补充信息或提示用户。

公开仓库不包含私有 Claude Code Key。使用者需要自行配置。

### 4. 插件系统

插件相关目录：

```text
baize/skills
src/routes/plugins.routes.js
src/services
client/desktop/local-runtime.cjs
```

当前主要插件方向：

- Jira：查询、草稿导入、确认创建、失败恢复。
- 企业微信：Webhook 接入结构。
- 本地插件桥：客户端确认权限后执行操作。

插件写操作设计上需要经过用户确认，避免未经授权修改外部系统。

### 5. Jira 工作流

Jira 创建流程的核心思想是：

1. 用户提出需求或上传需求文件。
2. 本地 Claude Code 分析并生成 Jira 草稿。
3. 客户端显示确认卡。
4. 用户确认后，Claude Code 才能调用 Jira 插件桥。
5. Claude Code 根据 Jira 返回结果分步处理：
   - 校验项目。
   - 获取创建元数据。
   - 修正 issueType、负责人、自定义字段。
   - 创建 Jira 单。
   - 遇到错误时给出中文恢复建议。
6. 客户端展示创建结果或恢复卡。

这样可以同时保留 AI 的动态判断能力和用户对写操作的最终控制权。

### 6. 记忆系统

记忆目录：

```text
baize/memory
```

白泽记忆分为两层：

- 浅层记忆：保存摘要、索引和快速召回信息。
- 深层记忆：保存完整分析、附件副本、详细上下文。

图片记忆的推荐流程是：

1. 客户端上传图片并保留本机路径。
2. 用户点击“加入记忆区”。
3. 客户端强制调用本机 Claude Code 做视觉分析。
4. 客户端只把脱敏后的分析结果发给服务端。
5. 服务端把图片副本和完整分析写入深层记忆。
6. 服务端把摘要和深层索引写入浅层记忆。

如果本机 Claude Code 不可用或本机路径缺失，则不写 fallback 记忆。

### 7. 逻辑断言系统

逻辑目录：

```text
baize/logic
```

逻辑断言用于保存项目规则、角色职责、流程约束和长期有效的判断依据。AI 在分析问题时可以结合这些规则，避免每次都从零开始。

### 8. 审计与权限

白泽的设计原则是：

- AI 可以分析和建议。
- 涉及外部系统写入时，需要用户确认。
- 插件调用结果需要可追踪。
- 错误应返回中文，并尽可能给出恢复路径。

这适合用于 Jira 创建、记忆写入、逻辑更新、外部系统操作等需要权限边界的场景。

## 适合的使用场景

- 个人本地 AI 工作台。
- 团队内部 AI 助手原型。
- Jira 需求拆分和创建助手。
- 企业微信/内部系统插件网关。
- 项目记忆与规则沉淀系统。
- Claude Code 桌面集成实验。
- AI 操作确认卡和权限边界研究。

## 不适合的使用方式

- 直接公开部署并使用默认配置。
- 把真实 API Key、Jira Token、上传文件提交到仓库。
- 跳过确认卡让 AI 静默写外部系统。
- 把运行时会话、用户聊天记录、记忆附件直接公开。

## 公开版与私有运行版的区别

公开版包含：

- 源码。
- 测试。
- 示例配置。
- 文档。
- 空的记忆结构。
- 插件和客户端代码。

公开版不包含：

- 私有 API Key。
- Jira Token。
- 内网地址。
- 用户聊天记录。
- 上传文件。
- 运行时操作记录。
- Windows 安装包。
- `node_modules` 依赖目录。

下载公开版后，需要执行 `npm install` 并填写自己的配置才能完整运行。
