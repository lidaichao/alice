## 1、使用的 IDE / Agent / 开发环境

本月主要使用以下 AI 工具和开发环境：

### IDE / 开发环境

- Cursor / VS Code 类 AI 编程环境
- Claude Code 命令行开发环境
- Node.js 服务端开发环境
- Electron 桌面端开发环境
- Unity 客户端开发与测试环境
- 本地 Windows 开发环境

### Agent / 工具

#### 白泽

白泽是本月重点建设和使用的 AI 工具，定位是研发协作入口和项目智能助手。当前主要能力包括：

- 接收自然语言需求并整理成研发任务。
- 解析项目上下文、需求表、附件摘要等信息。
- 辅助生成 Jira 创建草稿。
- 将需求内容拆解成模块、负责人、预估工时、验收标准等结构化信息。
- 支持客户端和服务端协作流程中的信息整理、需求归档和研发跟进。

本月白泽重点用于 JUMP 需求收集表的处理，将 Excel 表中的关卡需求池内容整理成 Jira 草稿，并梳理后续客户端、策划、UI、特效、模型等协作事项。

#### Unity 自动识图测试工具

Unity 自动识图测试工具是本人编写的自动化测试工具，用于客户端表现类需求的自动验证。

- 工程路径：`D:\zenghaorang\WorkSpace`
- 主要用途：通过图像识别判断 Unity 游戏画面是否符合预期。
- 适用场景：UI 通知、特效表现、镜头流程、角色显隐、Buff 表现、资源图标显示等。

本月该工具主要作为客户端表现验证方向的自动化建设重点，用于支撑后续 JUMP 关卡、搜打撤、通用关卡表现流程等需求的自动化回归。

## 2、用到的基础模型

本月主要使用以下基础模型或 AI 能力：

- Claude 系列模型
  - 用于代码理解、服务端逻辑开发、文档生成、需求拆解和方案设计。
- GPT 系列模型
  - 用于日常问答、方案补充、文本整理和部分代码辅助。
- IDE 内置 Agent 模型
  - 用于项目内代码搜索、局部改动、问题定位和重构建议。

在实际使用中，不同模型的分工大致如下：

- Claude 更适合长上下文理解、代码库级别分析、工具链设计和复杂文档生成。
- GPT 更适合快速问答、语言整理、邮件和总结类内容生成。
- IDE Agent 更适合结合当前工程进行局部代码修改和调试。

## 3、具体工作内容与规模

本月借助 AI 完成的工作主要分为以下几类。

### 3.1 白泽工具开发与优化

围绕白泽工具完成了多项开发和功能整理工作，包括：

- 服务端 Claude 聊天能力接入与上下文组织。
- Windows 桌面客户端 MVP 建设。
- 本地会话、配置、插件、附件、对话管理等服务能力建设。
- Jira 相关能力设计和服务拆分。
- 需求表解析、附件摘要、Jira 草稿生成等流程整理。
- 客户端与服务端协作边界梳理。

工作规模：中大型工具建设，涉及服务端、桌面端、配置、测试、文档和 Jira 流程多个模块。

### 3.2 JUMP 需求收集表处理

借助 AI 对 `JUMP需求收集表.xlsx` 中的【关卡】需求池进行整理，提取出通用关卡、关卡事件、搜打撤等模块需求，并生成 Jira 草稿方向。

包括的需求类型有：

- 关卡相机表现流程控制。
- 局内 UI 通知规则梳理与表现调整。
- 所有玩法新增词缀支持。
- 搜打撤召唤师技能钩爪特效与图标资源需求。
- Buff 隐身效果节点支持。
- 搜打撤怪物 AI 策划配置支持。

工作规模：中等规模需求整理，AI 主要承担表格摘要理解、任务拆分、Jira 草稿组织和文档生成工作。

### 3.3 Unity 自动识图测试工具建设

围绕 Unity 自动识图测试工具，梳理了其在客户端表现验证中的应用方式。

当前重点覆盖方向包括：

- UI 通知位置、大小、顺序和替换规则识别。
- 关卡相机 Timeline 表现流程验证。
- 角色隐身、半透明、可见范围和持续特效识别。
- 技能特效、图标、模型资源是否正确出现。
- 小局表演和镜头切换的关键帧验证。

工作规模：中等规模自动化测试工具建设，后续可扩展到更多客户端表现类回归测试。

### 3.4 日常研发辅助

AI 还用于日常研发中的以下工作：

- 代码结构理解和调用链梳理。
- Bug 定位和修复建议。
- 单元测试补充和测试失败分析。
- 配置文件和接口文档整理。
- 技术方案、需求文档、验收标准编写。
- 邮件、报告、阶段性总结生成。

工作规模：高频日常辅助，覆盖需求、开发、测试、文档多个环节。

## 4、与 AI 协作的方式

本月与 AI 的协作方式主要有以下几种。

### 4.1 需求到任务的结构化协作

对于需求表、聊天记录、策划描述等半结构化内容，先让 AI 提取关键信息，再转成研发可执行格式。

典型流程：

1. 输入需求原文或表格摘要。
2. AI 提取模块、开发内容、负责人、优先级、工时和验收点。
3. AI 生成 Jira 草稿或任务拆分。
4. 人工确认项目、负责人和拆分方式。
5. 白泽执行后续创建或归档流程。

### 4.2 代码开发中的 Agent 协作

在代码开发中，主要采用“人定方向，AI 执行细节”的方式：

- 人负责明确目标、边界和验收方式。
- AI 负责搜索代码、定位文件、生成修改方案。
- 修改后由人检查关键逻辑。
- 再通过测试和实际运行验证结果。

这种方式可以减少重复搜索和样板代码编写时间，但核心设计和最终验收仍由人把控。

### 4.3 自动化测试协作

对于 Unity 自动识图测试工具，AI 主要辅助：

- 梳理哪些需求适合自动化。
- 设计测试用例命名和覆盖范围。
- 总结图像识别测试所需资产。
- 将 Jira 需求和自动化测试用例建立关联。

实际图像识别逻辑、测试场景稳定性和 Unity 工程集成仍需要结合项目环境人工验证。

### 4.4 文档和总结协作

本月大量使用 AI 辅助生成文档初稿，包括：

- 工具说明文档。
- Jira 描述模板。
- 需求落地流程。
- 自动化测试方案。
- 月度使用总结。

AI 负责提高初稿产出效率，人负责调整内容准确性和项目语境。

## 5、本月收获

### 5.1 AI 更适合做研发流程的“连接器”

相比单纯写代码，AI 在需求整理、任务拆分、上下文归纳、Jira 草稿生成、验收标准补充等环节的价值更稳定。白泽工具也是基于这个方向建设的，希望把 AI 从单次问答升级为项目研发流程中的协作入口。

### 5.2 长上下文对项目工具非常重要

在白泽场景中，AI 需要理解项目背景、历史对话、附件、配置、Jira 流程和用户意图。只有具备足够上下文，生成的任务草稿和建议才不会脱离实际。

### 5.3 AI 可以显著降低文档和流程整理成本

需求表、验收标准、Jira 模板、测试方案这类内容，以前需要较多人工整理。本月通过 AI 可以快速形成结构化初稿，再由人工修订，效率提升明显。

### 5.4 自动化测试需要和需求管理更早结合

Unity 自动识图测试工具如果只在开发完成后再补用例，容易遗漏关键场景。更好的方式是在 Jira 创建阶段就写入“是否适合自动化覆盖”和“建议验证点”，让自动化测试从需求阶段就介入。

## 6、遇到的问题 / “发狂” 时刻

### 6.1 AI 对不完整附件内容容易过度推断

当 Excel 文件只提供摘要而不是完整原始内容时，AI 有时会根据可见内容生成看似完整的结论。实际使用中必须明确标注“仅基于当前可见摘要”，避免误创建 Jira 或遗漏隐藏行。

### 6.2 Jira 字段映射需要人工兜底

AI 可以生成 Jira 草稿，但项目 Key、Issue Type、人员账号映射、组件归属等信息如果缺失，不能直接自动创建。否则可能出现任务归属错误或项目流转错误。

### 6.3 代码 Agent 偶尔会改得过多

在复杂工程中，AI 有时会为了“顺手优化”扩大改动范围。实际协作时需要明确限制：只改当前任务需要的内容，不做额外重构。

### 6.4 识图自动化对环境稳定性要求高

Unity 自动识图测试工具虽然适合验证表现类需求，但对分辨率、画质、场景状态、特效随机性和截图时机比较敏感。如果测试环境不稳定，识别结果容易波动。

### 6.5 AI 生成内容需要项目语境校正

AI 生成的文档和方案通常结构较好，但术语、团队流程、负责人、真实路径、实际工具能力等细节必须人工确认，否则容易出现“格式正确但内容不完全准确”的问题。

## 7、整体使用感受

整体来看，本月 AI 对研发效率提升比较明显，尤其是在以下方面：

- 需求整理速度更快。
- Jira 草稿生成更规范。
- 文档和总结产出成本更低。
- 代码搜索、问题定位、测试补充效率提升。
- 自动化测试方案可以更早进入需求讨论。

白泽工具的价值在于把 AI 能力沉淀到项目流程中，而不是停留在单次对话。它可以作为需求入口、研发助手和 Jira 协作工具，帮助团队把零散输入转成可追踪任务。

Unity 自动识图测试工具的价值在于补足客户端表现类需求的自动化验证能力。很多 UI、特效、镜头、隐身表现问题，传统单元测试很难覆盖，而识图测试可以更接近真实玩家画面结果。

总体感受是：AI 适合承担“整理、搜索、生成初稿、辅助执行”的工作，但关键判断、流程确认和最终验收仍需要人负责。后续如果能把白泽、Jira 和 Unity 自动化测试进一步打通，AI 在研发流程中的价值会更明显。
