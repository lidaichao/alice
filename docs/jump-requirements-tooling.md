## 目标

通过工具化流程把 Excel 需求池转成可追踪、可执行、可验证的研发任务：

1. 使用白泽解析需求表，生成 Jira 创建草稿。

## 工具一：白泽

白泽是需求流转和研发协作入口，主要负责把非结构化或半结构化输入转成可执行事项。

### 当前作用

- 接收并解析 `JUMP需求收集表.xlsx`。
- 从表格摘要中提取有效需求行。
- 按模块和开发内容生成 Jira 草稿。
- 将策划内容、拆分、负责人、工时、提出时间等字段整理为 Jira 描述。
- 在创建前生成确认卡，避免误创建或重复创建 Jira 单。

### 建议流程

1. 上传需求表到白泽。
2. 白泽读取工作表【关卡】需求池。
3. 过滤空行和无效行。
4. 按“关卡模块 + 开发内容”生成 Jira 标题。
5. 按“拆分”字段判断任务涉及的协作方。
6. 按“预估工时-必填”写入 Jira 描述。
7. 对未配置项目 Key、任务类型或负责人映射的草稿进行阻断提示。
8. 负责人确认后创建 Jira 单，并回写 Jira 链接。

### Jira 草稿字段建议

| Jira 字段 | 来源字段 | 说明 |
| --- | --- | --- |
| 标题 | 关卡模块 + 开发内容 | 示例：【搜打撤】Buff 隐身效果节点支持 |
| 描述 | 需求说明、备注 | 保留策划原始描述，避免信息丢失 |
| 优先级 | 优先级 | 极高、中等映射到 Jira 优先级 |
| 负责人 | 处理人 | 需要映射到 Jira 账号 |
| 参与方 | 拆分 | 用于判断客户端、策划、UI、动效、特效、模型等协作范围 |
| 预估 | 预估工时-必填 | 写入描述或 Jira 工时字段 |
| 策划负责人 | 策划负责人 | 作为需求验收联系人 |
| 提出时间 | 提出时间 | 用于追踪需求来源时间 |

## 工具二：Unity 自动化识图测试工具

Unity 自动化测试工具是用于客户端表现验证的自动化能力，核心是通过图像识别判断游戏画面是否符合预期。

工作地址：`D:\zenghaorang\WorkSpace`

### 当前定位

该工具适合覆盖以下类型需求：

- UI 通知位置、大小、显示优先级、替换规则验证。
- 关卡相机流程表现验证。
- 角色半透明、隐身、可见范围等表现状态验证。
- 特效、图标、模型资源是否正确出现。
- 小局表演、镜头切换、Timeline 表演结果验证。

### 与本次需求的对应关系

| 需求方向 | 可自动化验证点 | 建议验证方式 |
| --- | --- | --- |
| 关卡相机表现流程控制 | 镜头是否切换到目标位置，Timeline 阶段是否正确 | 识别关键帧画面、相机视角标志物或 UI 状态 |
| 局内 UI 通知规则梳理与表现调整 | 通知出现顺序、位置、尺寸、替换关系 | 识别通知图块位置和出现时序 |
| 所有玩法新增词缀支持 | 词缀 UI、战斗内表现或状态提示是否出现 | 识别词缀图标、文本区域或状态标识 |
| 召唤师技能钩爪特效与图标资源需求 | 钩爪特效、眼睛图标、感叹号图标是否正确显示 | 识别特效关键画面和 UI 图标模板 |
| Buff 隐身效果节点支持 | 半透明表现、敌方不可见、范围内可见、持续特效出现/消失 | 对比不同阵营视角截图，识别角色透明度和特效状态 |
| 搜打撤怪物 AI 策划配置 | 怪物行为状态是否符合配置 | 结合录屏关键帧识别怪物位置、行为阶段和状态 UI |

### 测试资产建议

为了让识图测试稳定，需要为每类需求准备以下资产：

- 标准测试关卡或测试场景。
- 固定分辨率和固定画质配置。
- 关键 UI 图标模板，例如眼睛、感叹号、通知样式。
- 特效关键帧截图模板。
- 角色隐身前、隐身中、被发现、未被发现的对照截图。
- 每条自动化用例对应的 Jira 链接。

### 用例命名建议

建议统一用例命名格式：

```text
[JiraKey]_[模块]_[验证目标]
```

示例：

```text
JUMP-123_搜打撤_Buff隐身范围内可见
JUMP-124_通用关卡_UI通知替换规则
JUMP-125_通用关卡_关卡相机Timeline流程
```

## 推荐落地流程

### 1. 需求录入

策划继续维护 `JUMP需求收集表.xlsx`，保证每条需求至少包含：

- 关卡模块。
- 开发内容。
- 需求说明。
- 优先级。
- 拆分。
- 处理人。
- 预估工时。
- 策划负责人。

### 2. 白泽解析

白泽读取表格后生成 Jira 草稿。对于字段缺失或无法映射的内容，不直接创建 Jira，而是进入确认流程。

重点检查：

- 是否有重复需求。
- 是否已有 Jira 链接。
- 是否缺少项目 Key。
- 是否需要拆子任务。
- 是否需要补充验收标准。

### 3. Jira 创建

确认后由白泽创建 Jira 单。创建成功后建议将 Jira 链接回写到需求表，避免重复创建。

每个 Jira 单建议包含：

- 需求原文。
- 任务拆分。
- 负责人。
- 工时预估。
- 验收标准。
- 自动化测试建议。

### 4. 开发与联调

客户端、策划配置、UI、动效、特效、模型等协作方按 Jira 任务推进。

对客户端表现类需求，需要同步整理自动化测试入口和测试场景。

### 5. Unity 识图自动化验证

当需求进入可验证阶段后，在 `D:\zenghaorang\WorkSpace` 下使用 Unity 自动化识图测试工具补充回归用例。

优先覆盖：

1. 高优先级需求。
2. 容易回归出错的表现流程。
3. 人工验证成本高的 UI 顺序、位置、镜头、特效类需求。
4. 后续版本会反复改动的核心玩法能力。

### 6. 验收与回写

测试完成后，将结果回写到 Jira 或需求表：

- 策划验收状态。
- QA 验收结果。
- 自动化用例是否已覆盖。
- 失败截图或录屏路径。
- 问题单链接。

## 风险与注意事项

- 当前附件摘要只展示了部分表格内容，完整 xlsx 仍需要由白泽或服务端解析后再批量创建 Jira。
- “项目 Key 未设置”时不应直接创建 Jira。
- 表格中空行、半截行和重复日期行需要过滤。
- 美术、UI、特效、模型类需求可能不应全部归到客户端任务，需要按团队流程确认拆分方式。
- 识图自动化测试对分辨率、画质、场景稳定性敏感，需要统一运行环境。
- 对隐身、透明、特效消失这类表现，需要准备正反样本，避免只验证“出现”不验证“消失”。
